# hitl_mcp · 实现思路

> 作品集编号：**W6**（自研 MCP）  
> 全称：Human-in-the-Loop Approval Gate MCP  
> HITL = Human-in-the-Loop（人在回路）  
> 更新日期：2026-07-14  
> 状态：v0.2 — 内置策略 + assess_and_gate 自动开单 + 外置 hitl-gate Skill  

---

## 1. 一句话产品定义

给 Agent 的高风险工具调用加一道**可审计的审批闸门**：  
Agent 只能申请，人在**本地网页面板**批准/拒绝后，Agent 才能继续；拒绝或过期则必须停止。

---

## 2. 要解决的问题

| 没有门 | 有 hitl_mcp |
|--------|-------------|
| Agent 直接删文件 / 推代码 / 调写接口 | 先挂起，等人确认 |
| 「聊天里回一句 OK」难审计、易被注入绕过 | 有 ticket、状态机、参数哈希、过期 |
| 每个 Agent 框架自己做 interrupt | 做成标准 **MCP**，Cursor / 自研 Client 都能接 |

**非目标（刻意不做）**

- 不做内容审核 / 脏话过滤（避免与 `censor` 类产品混淆）  
- 不做通用 GitHub/浏览器 MCP  
- 不做公网多租户审批 SaaS（Demo 只绑本机）  

---

## 3. 总体架构

```text
┌─────────────────────┐         MCP (stdio)          ┌──────────────────────┐
│  Agent Client       │  request_approval            │  hitl_mcp Server     │
│  (Cursor / 自研)    │  get_approval_status         │                      │
│                     │  list_pending                │  Tool handlers       │
│                     │ ◄──────────────────────────► │         │            │
└─────────────────────┘                              │         ▼            │
                                                     │  SQLite / JSON store │
┌─────────────────────┐         HTTP 本机            │         ▲            │
│  审批面板 Web UI    │  GET list / POST resolve     │         │            │
│  127.0.0.1:8787     │ ◄──────────────────────────► │  同一存储层          │
└─────────────────────┘                              └──────────────────────┘
```

**两条入口，一份真相**

- **MCP tools**：给 Agent 用（申请、查状态、列待审）  
- **本地网页**：给人用（看卡片、批准/拒绝）  
- 存储只有一份，避免状态不一致  

可选加分：同存储上的 CLI（`hitl pending | approve | reject`），主视觉仍以网页为准。

## 4. 核心状态机

```text
                 request_approval
        (无) ──────────────────────► pending
                                       │
                    ┌──────────────────┼──────────────────┐
                    │                  │                  │
                    ▼                  ▼                  ▼
                approved           rejected            expired
                    │                  │                  │
                    ▼                  ▼                  ▼
            Agent 可继续执行      Agent 必须停止     需重新申请
            （建议先 dry-run）
```

| 状态 | 含义 | 面板视觉建议 |
|------|------|----------------|
| `pending` | 等待人处理 | 橙/红卡片 + 倒计时 |
| `approved` | 已批准 | 绿色完成 |
| `rejected` | 已拒绝 | 灰色 + 原因 |
| `expired` | 超时未处理 | 灰色过期 |
| `cancelled` | （可选）发起方取消 | 灰色 |

**规则**

- `pending` 之外不可再改成 `approved`（防重复点）  
- 过期后 Agent 再查必须得到 `expired`，不能当默认同意  
- 批准时校验**参数哈希**：执行前若参数被改，视为无效，需重新申请

## 5. MCP Tools 设计（MVP：4 个）

### 5.1 `request_approval`

Agent 发起审批。

**入参（建议）**

| 字段 | 类型 | 说明 |
|------|------|------|
| `action` | string | 动作名，如 `delete_files` / `git_force_push` |
| `summary` | string | 给人看的一句话 |
| `params` | object | 拟执行参数（将被锁定） |
| `risk` | enum | `low` \| `medium` \| `high` \| `critical` |
| `ttl_seconds` | number | 默认 600；超时 → expired |
| `requester` | string | 可选：Skill 名 / Agent 名 |

**出参**

```json
{
  "ticket_id": "apr_7f3a9c",
  "status": "pending",
  "expires_at": "2026-07-14T21:00:00+08:00",
  "panel_url": "http://127.0.0.1:8787/tickets/apr_7f3a9c",
  "params_hash": "sha256:..."
}
```

### 5.2 `get_approval_status`

**入参**：`ticket_id`  
**出参**：`status`、`decision_reason`、`decided_at`、`params_hash`、是否仍有效  

Agent 在 pending 时应**轮询或提示用户去面板**，不要假装已批准。

### 5.3 `list_pending`

列出当前待审单（调试与 Skill 自检用）。可带 `risk` 过滤。

### 5.4 不通过 MCP 暴露的 `resolve`（给人）

批准/拒绝走 **Web API**（或 CLI），避免 Agent 自己调 `approve` 绕过 HITL。

若必须提供 MCP tool，应命名为高风险且文档写清「仅供人/运维」，Demo 阶段**默认不注册给 Agent**。

## 6. 本地网页审批面板（主视觉）

### 6.1 为什么用网页而不是只靠聊天确认

- 聊天回「可以」：**弱 HITL**，无工单、难审计  
- 面板点批准：**强 HITL**，状态机 + 哈希 + 过期  
- 契合前端背景，录屏分屏效果好  

### 6.2 页面结构（最小 2 页）

1. **列表页** `/`  
   - Tab：待审 / 已处理 / 已过期  
2. **详情页** `/tickets/:id`  
   - 动作、摘要、参数 JSON、风险、倒计时  
   - 按钮：`批准并继续` / `拒绝`  
   - 拒绝原因输入框  

### 6.3 卡片字段（必有）

- ticket_id、risk、action、summary  
- params 摘要（可展开全文）  
- params_hash、expires 倒计时  
- requester（Skill/Agent）  
- 批准 / 拒绝操作区  

### 6.4 Demo 录屏分镜

```text
左：Cursor 对话出现 ticket + ⏳ pending
右：打开 127.0.0.1:8787，卡片已在待审
右：点「批准」→ 卡片变绿
左：Agent get_approval_status → approved → 继续（dry-run）
再来一遍：点「拒绝」→ Agent 明确停止并读出原因
```

### 6.5 安全（Demo 级）

- 只监听 `127.0.0.1`，不绑 `0.0.0.0`  
- 可选：面板请求头带本地 token（环境变量）  
- 生产再谈 SSO；面试口述边界即可

## 7. 数据模型（建议 SQLite）

表 `approvals`：

| 列 | 说明 |
|----|------|
| `ticket_id` | 主键 |
| `action` | 动作名 |
| `summary` | 摘要 |
| `params_json` | 锁定参数 |
| `params_hash` | sha256 |
| `risk` | 等级 |
| `status` | pending/approved/rejected/expired/cancelled |
| `requester` | 可选 |
| `created_at` | |
| `expires_at` | |
| `decided_at` | 可空 |
| `decision_reason` | 拒绝原因等 |
| `decided_by` | 如 `web` / `cli` |

过期：读时若 `now > expires_at && status==pending`，惰性更新为 `expired`。

审计：MVP 用同一表即可；加分再拆 `audit_log`（每次状态变更一行）。

## 8. 推荐技术栈（易实现优先）

| 层 | 建议 | 备选 |
|----|------|------|
| 语言 | **TypeScript (Node)** | Python |
| MCP SDK | `@modelcontextprotocol/sdk` | 官方 Python SDK |
| 传输 | **stdio**（Cursor 最顺） | 加分再 SSE |
| 存储 | **better-sqlite3** 或 JSON 文件 | |
| Web | 同进程 `http` + 简单 HTML/CSS，或 Express + 静态页 | FastAPI + Jinja |
| 前端 | 无构建单页即可（状态少） | 若想炫再上轻量 React |

原则：**一个 repo、一个进程可同时起 MCP + 可选内嵌 Web**；或 MCP 与 Web 两个进程共 SQLite 文件。

## 10. 第一个 Demo 场景（建议）

**场景名**：安全清理仓库中的密钥文件（dry-run）

1. Skill 指示：匹配 `**/.env` 的删除必须先 HITL  
2. Agent 调 `request_approval`（params 含 glob 与文件列表）  
3. 人在面板批准或拒绝  
4. 若批准：执行器**只打印**将删除路径，不真删（面试更安全）  
5. 若拒绝：Agent 停止并引用 `decision_reason`  

备选场景：假「发布到生产」按钮、假「强制 push」——同样只 dry-run。

## 11. 与 W5 Skills 的联调

理想联演示：

> **Skill = 流程 SOP；hitl_mcp = 能力闸门。**

Skill 正文写明：

1. 识别是否高风险  
2. 必须调用 `request_approval`  
3. 轮询或提示用户打开 `panel_url`  
4. 仅当 `approved` 才进入执行步骤  
5. `rejected` / `expired` 则结束并汇报  

面试对比句：

> Rules 管始终遵守的约束；Skills 管某类任务怎么做；MCP（hitl）管外部/敏感能力如何被授权。

## 12. 验收标准（对齐作品集 Done）

| 项 | 标准 |
|----|------|
| 可跑 | README：装依赖 → 配 Cursor → 15 分钟内调通 |

## 14. 面试口述要点（备查）

1. **为何不是聊天确认**：要工单、哈希、过期、审计。  
2. **为何 resolve 不给 Agent**：防止自批自过。  
3. **和框架 interrupt 关系**：运行时机制 vs 可复用 MCP 能力。  
4. **生产升级**：SSO、推送通知、与真实工具代理链式放行。  
5. **红线**：Demo dry-run；不绑公网；不碰公司真实密钥。
